## 1. 문서 개요

### 1.1 목적

본 문서는 **Desktop Organizer** (이하 "제품")의 MVP(Minimum Viable Product) 범위에 해당하는 모든 기능을 정의한다. 이 문서는 개발, QA, 기획 전 구성원이 공유하는 **단일 진실 공급원(Single Source of Truth)**이며, 코드 구현에 앞서 기능의 정의와 범위를 확정하는 것을 목적으로 한다.

### 1.2 제품 개요

Desktop Organizer는 Windows 바탕화면 아이콘을 **자동으로 분류·관리**하는 생산성 프로그램이다. 사용자가 정의한 **Container(컨테이너)**와 **Rule(규칙)**을 기반으로 바탕화면을 지속적으로 감시하고, 새로 생성되거나 변경된 파일을 실시간으로 정리한다.

### 1.3 핵심 설계 원칙

| 원칙 | 설명 |
|------|------|
| **비파괴 원칙** | 아이콘의 위치(좌표)만 변경하며, 실제 파일은 절대 이동·복사·삭제하지 않는다. |
| **비침습 원칙** | Windows 기본 시스템 아이콘은 자동 정리 대상에서 기본 제외한다. |
| **지속성 원칙** | 모든 설정과 레이아웃은 저장·복원이 가능해야 한다. |
| **실시간성 원칙** | 바탕화면 변경을 실시간으로 감지하고 처리한다. |
| **규칙 우선성** | 모든 자동화는 사용자가 정의한 Rule에 의해서만 동작한다. |

### 1.4 MVP 범위 기준

- Windows 10 / Windows 11 지원
- 단일 모니터 기준 (다중 모니터는 Future)
- 단일 사용자 로컬 환경 (Cloud Sync는 Future)
- AI 자동 분류 기능 제외 (Future)

## 2. 용어 정의

| 용어 | 정의 |
|------|------|
| **아이콘(Icon)** | 바탕화면에 표시되는 파일, 폴더, 바로가기, 시스템 항목의 시각적 표현 단위 |
| **Container** | 바탕화면 위에 사용자가 정의한 가상의 구역(영역). 아이콘을 그룹화하기 위한 논리적·시각적 단위 |
| **Rule** | 특정 조건을 만족하는 아이콘을 지정한 Container로 자동 배치하는 규칙 |
| **조건(Condition)** | Rule이 아이콘에 적용될지 여부를 판단하는 기준 (파일명 패턴, 확장자, 날짜 등) |
| **액션(Action)** | Rule이 매칭되었을 때 실행되는 동작 (특정 Container로 배치 등) |
| **Layout** | 특정 시점의 모든 Container 위치·크기·스타일과 아이콘 배치 상태의 스냅샷 |
| **기본 아이콘** | Windows가 기본 제공하는 시스템 아이콘 (내 컴퓨터, 휴지통, 네트워크 등) |
| **정렬(Sort)** | Container 내 아이콘을 특정 기준(이름, 날짜, 종류 등)으로 순서화하는 행위 |
| **설정(Settings)** | 프로그램 전역 동작을 제어하는 사용자 구성값 |
| **감시(Watch)** | 바탕화면 디렉터리의 파일 시스템 이벤트를 실시간으로 탐지하는 동작 |

### F-001 바탕화면 읽기

| 항목 | 내용 |
|------|------|
| **기능 ID** | F-001 |
| **기능명** | 바탕화면 읽기 |
| **설명** | 프로그램 시작 시 Windows 바탕화면에 존재하는 모든 아이콘의 정보(파일명, 경로, 좌표, 유형)를 수집한다. |
| **목적** | 이후 모든 분류·정렬·자동화 기능의 기반 데이터를 확보한다. |
| **우선순위** | Must Have |
| **의존 기능** | 없음 |

#### 상세 동작

1. 프로그램이 시작되면 Windows Shell API를 통해 바탕화면 폴더 경로를 조회한다.
   - 사용자 바탕화면: `%USERPROFILE%\Desktop`
   - 공용 바탕화면: `%PUBLIC%\Desktop`
2. 두 경로에 존재하는 모든 항목을 열거한다.
3. 각 항목에 대해 아래 속성을 수집한다.

| 속성 | 설명 |
|------|------|
| 파일명 | 확장자 포함 전체 이름 |
| 절대 경로 | 파일 시스템 상의 전체 경로 |
| 아이콘 유형 | 파일 / 폴더 / 바로가기(.lnk) / 시스템 항목 |
| 현재 바탕화면 좌표 | (x, y) 픽셀 좌표 |
| 파일 확장자 | 없을 경우 빈 문자열 |
| 파일 생성일시 | UTC 기준 |
| 파일 수정일시 | UTC 기준 |
| 시스템 아이콘 여부 | Boolean |

4. 수집된 데이터는 내부 아이콘 목록(In-memory)으로 보관한다.
5. 수집 완료 후 F-002(파일 분류)를 트리거한다.

#### 입력

| 항목 | 값 |
|------|-----|
| 트리거 | 프로그램 시작 이벤트 |
| 외부 의존 | Windows Shell API (SHGetSpecialFolderPath, IShellFolder) |

#### 출력

| 항목 | 값 |
|------|-----|
| 아이콘 목록 | 수집된 모든 아이콘의 속성 집합 (In-memory) |
| 상태 | 성공 / 실패 |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| 바탕화면 경로 접근 권한 없음 | 오류 로그 기록 후 프로그램 종료 또는 사용자에게 알림 |
| Shell API 호출 실패 | 최대 3회 재시도 후 실패 시 오류 로그 기록 |
| 바탕화면 아이콘이 0개 | 정상 처리로 간주, 빈 목록 반환 |
| 개별 아이콘 속성 조회 실패 | 해당 아이콘만 건너뛰고 나머지는 계속 처리, 경고 로그 기록 |

#### 완료 기준 (Acceptance Criteria)

- [ ] 프로그램 시작 시 바탕화면의 모든 아이콘이 수집된다.
- [ ] 사용자 바탕화면과 공용 바탕화면을 모두 읽는다.
- [ ] 수집된 각 아이콘은 위 8가지 속성을 모두 포함한다.
- [ ] 아이콘 수집 중 일부 항목이 실패해도 나머지 항목은 정상 수집된다.
- [ ] 아이콘이 없는 바탕화면에서도 오류 없이 동작한다.

#### 비고

- 아이콘 좌표 수집은 Windows가 제공하는 실제 픽셀 좌표를 기준으로 한다.
- DPI 스케일링 환경에서의 좌표 처리 방식은 별도 기술 결정 필요.

### F-002 파일 분류

| 항목 | 내용 |
|------|------|
| **기능 ID** | F-002 |
| **기능명** | 파일 분류 |
| **설명** | 수집된 아이콘 목록을 미리 정의된 기준에 따라 유형별로 분류한다. |
| **목적** | Rule 매칭 및 Container 배치의 전처리 단계로, 각 아이콘이 어떤 유형에 속하는지 판별한다. |
| **우선순위** | Must Have |
| **의존 기능** | F-001 (바탕화면 읽기), F-003 (기본 아이콘 제외) |

#### 상세 동작

1. F-001에서 수집된 아이콘 목록을 입력으로 받는다.
2. F-003의 기본 아이콘 목록을 참조하여 시스템 아이콘을 제외한다.
3. 남은 아이콘을 아래 분류 기준으로 유형을 부여한다.

| 분류 유형 | 판별 기준 |
|-----------|-----------|
| 문서 | .doc, .docx, .xls, .xlsx, .ppt, .pptx, .pdf, .txt, .hwp 등 |
| 이미지 | .jpg, .jpeg, .png, .gif, .bmp, .webp, .svg, .psd, .ai 등 |
| 동영상 | .mp4, .avi, .mov, .mkv, .wmv, .flv 등 |
| 음악 | .mp3, .wav, .flac, .aac, .ogg, .m4a 등 |
| 압축파일 | .zip, .rar, .7z, .tar, .gz 등 |
| 실행파일 | .exe, .msi, .bat, .cmd, .ps1 등 |
| 바로가기 | .lnk 확장자 |
| 폴더 | 파일 시스템 디렉터리 |
| 기타 | 위 분류에 해당하지 않는 모든 항목 |

4. 각 아이콘에 분류 유형(Type Label)을 속성으로 추가한다.
5. 분류 결과는 내부 목록에 반영하고 Rule 매칭(F-012~F-015)에 활용할 수 있도록 준비한다.

#### 입력

| 항목 | 값 |
|------|-----|
| 아이콘 목록 | F-001에서 수집된 아이콘 속성 집합 |
| 기본 아이콘 목록 | F-003에서 정의된 제외 목록 |

#### 출력

| 항목 | 값 |
|------|-----|
| 분류된 아이콘 목록 | 유형(Type Label)이 부여된 아이콘 속성 집합 |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| 확장자가 없는 파일 | "기타" 유형으로 분류 |
| 확장자는 있으나 목록에 없는 경우 | "기타" 유형으로 분류 |
| 분류 목록 파일 로딩 실패 | 내장 기본 분류 목록으로 대체, 경고 로그 기록 |

#### 완료 기준 (Acceptance Criteria)

- [ ] 수집된 모든 아이콘에 유형이 부여된다.
- [ ] 기본 아이콘은 분류 대상에서 제외된다.
- [ ] 알 수 없는 확장자는 "기타"로 분류된다.
- [ ] 분류 유형 목록은 설정을 통해 확장 가능한 구조여야 한다.

#### 비고

- 분류 기준(확장자 목록)은 별도 설정 파일로 관리하며 사용자가 직접 수정할 수 있어야 한다.
- 동일 확장자가 복수 유형에 중복 등록된 경우, 목록 상 먼저 정의된 유형을 우선 적용한다.

### F-003 기본 아이콘 제외

| 항목 | 내용 |
|------|------|
| **기능 ID** | F-003 |
| **기능명** | 기본 아이콘 제외 |
| **설명** | Windows가 기본 제공하는 시스템 아이콘을 자동 정리 대상에서 제외한다. |
| **목적** | 사용자가 의도하지 않은 시스템 아이콘의 이동을 방지하여 시스템 안정성을 보장한다. |
| **우선순위** | Must Have |
| **의존 기능** | F-001 (바탕화면 읽기) |

#### 상세 동작

1. 프로그램이 초기화될 때 기본 제외 아이콘 목록을 로딩한다.
2. 기본 제외 목록에는 아래 항목이 포함된다.

| 항목 | 설명 |
|------|------|
| 내 PC (This PC) | CLSID 기반 시스템 아이콘 |
| 휴지통 (Recycle Bin) | CLSID 기반 시스템 아이콘 |
| 내 문서 (Documents) | CLSID 기반 시스템 아이콘 |
| 네트워크 (Network) | CLSID 기반 시스템 아이콘 |
| 제어판 (Control Panel) | CLSID 기반 시스템 아이콘 |

3. 제외 여부는 파일명 외에 CLSID(클래스 식별자) 기반으로도 판별한다.
4. 사용자는 설정 화면에서 기본 제외 목록을 확인하고 개별 항목을 정리 대상에 포함시킬 수 있다.
5. 사용자가 직접 파일이나 폴더를 추가 제외 목록에 등록할 수 있다.
6. 제외된 아이콘은 Rule 매칭 및 자동 정렬 대상에서 완전히 제외된다.

#### 입력

| 항목 | 값 |
|------|-----|
| 아이콘 목록 | F-001에서 수집된 전체 목록 |
| 기본 제외 목록 | 내장 시스템 아이콘 식별자 목록 |
| 사용자 추가 제외 목록 | 사용자가 설정에서 지정한 추가 제외 항목 |

#### 출력

### F-004 Container 생성

| 항목 | 내용 |
|------|------|
| **기능 ID** | F-004 |
| **기능명** | Container 생성 |
| **설명** | 사용자가 바탕화면 위에 새로운 Container를 생성한다. Container는 아이콘을 그룹화하는 가상의 영역이다. |
| **목적** | 사용자가 바탕화면을 논리적으로 구역화하여 아이콘을 체계적으로 관리할 수 있게 한다. |
| **우선순위** | Must Have |
| **의존 기능** | F-001 (바탕화면 읽기) |

#### 상세 동작

1. 사용자가 Container 생성을 요청한다 (메뉴, 단축키 또는 바탕화면 우클릭).
2. 시스템은 새로운 Container 객체를 생성하고 아래 기본값을 적용한다.

| 속성 | 기본값 |
|------|--------|
| Container ID | 고유 식별자 (UUID) |
| Container 이름 | "새 Container" (생성 순번 자동 부여) |
| 위치 (x, y) | 현재 마우스 커서 위치 또는 바탕화면 중앙 |
| 크기 (width, height) | 기본값 (설정에서 변경 가능) |
| 배경색 | 기본 반투명 색상 |
| 테두리 스타일 | 기본 실선 |
| 정렬 방식 | 이름 기준 오름차순 |
| 연결된 Rule | 없음 |

3. 생성된 Container는 바탕화면 오버레이 레이어에 즉시 표시된다.
4. Container 이름은 생성 직후 편집 가능 상태로 활성화된다.
5. Container 정보는 설정 파일에 저장된다 (F-018 연동).

#### 입력

| 항목 | 값 |
|------|-----|
| 트리거 | 사용자 액션 (메뉴 선택, 단축키) |
| 생성 위치 | 마우스 좌표 또는 기본 위치 |

#### 출력

| 항목 | 값 |
|------|-----|
| 새 Container | 고유 ID를 가진 Container 객체 |
| 시각적 표현 | 바탕화면에 Container 영역 표시 |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| 기존 Container와 위치 완전 중복 | 신규 Container를 약간 오프셋하여 배치 |
| 최대 Container 수 초과 | 사용자에게 경고 메시지 표시 (최대 수는 설정에서 정의) |
| 저장 실패 | 생성은 유지하되 오류 로그 기록 및 사용자 알림 |

#### 완료 기준 (Acceptance Criteria)

- [ ] Container가 생성되면 바탕화면에 즉시 시각적으로 표시된다.
- [ ] 각 Container는 고유 ID를 가진다.
- [ ] Container 이름은 생성 직후 즉시 편집 가능하다.
- [ ] 생성된 Container는 설정 파일에 저장된다.
- [ ] 생성된 Container는 프로그램 재시작 후에도 유지된다.

#### 비고

- Container는 바탕화면 위에 오버레이로 표시되며 실제 파일 시스템과 무관하다.
- Container의 최대 생성 수는 초기 버전 기준 설정 필요 (예: 50개).

- [ ] 정렬 방식 변경 시 즉시 재정렬된다.

- [ ] 수정 사항이 설정 파일에 저장된다.
- [ ] Undo 기능으로 직전 상태로 되돌릴 수 있다.

### F-006 Container 삭제

| 항목 | 내용 |
|------|------|
| **기능 ID** | F-006 |
| **기능명** | Container 삭제 |
| **설명** | 사용자가 선택한 Container를 삭제한다. Container만 제거되며 내부 아이콘(실제 파일)은 영향을 받지 않는다. |
| **목적** | 불필요해진 Container를 제거하여 바탕화면을 정리할 수 있게 한다. |
| **우선순위** | Must Have |
| **의존 기능** | F-004 (Container 생성), F-018 (설정 저장) |

#### 상세 동작

1. 사용자가 삭제할 Container를 선택하고 삭제 액션을 실행한다.
2. 삭제 확인 다이얼로그를 표시한다.
   - 다이얼로그에는 "Container 내 아이콘은 삭제되지 않습니다"라는 안내 문구를 포함한다.
3. 사용자가 확인 시:
   1. Container와 연결된 Rule의 Container 참조를 제거한다 (Rule 자체는 삭제하지 않음).
   2. Container 내 아이콘의 위치를 바탕화면 기본 위치로 초기화하거나 현재 위치 유지.
   3. Container 객체를 제거한다.
   4. 설정 파일에서 해당 Container 정보를 삭제한다.
4. 사용자가 취소 시: 아무 동작도 하지 않는다.
5. 삭제 동작은 Undo 스택에 저장된다.

#### 입력

| 항목 | 값 |
|------|-----|
| 대상 Container ID | 삭제할 Container의 고유 식별자 |
| 사용자 확인 | 확인 또는 취소 |

#### 출력

| 항목 | 값 |
|------|-----|
| 삭제 결과 | 성공 / 취소 / 실패 |
| 해제된 아이콘 | Container에서 해제된 아이콘 목록 (실제 파일 변경 없음) |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| 삭제 중 설정 파일 쓰기 실패 | 오류 로그 기록, 사용자 알림, Container는 화면에서 제거 |
| 연결된 Rule 업데이트 실패 | 오류 로그 기록, Rule은 그대로 유지하고 사용자 수동 처리 안내 |

#### 완료 기준 (Acceptance Criteria)

- [ ] Container 삭제 시 확인 다이얼로그가 표시된다.
- [ ] 다이얼로그에 아이콘은 삭제되지 않는다는 안내가 포함된다.
- [ ] Container 삭제 후 실제 파일은 변경되지 않는다.
- [ ] 삭제된 Container 정보는 설정 파일에서 제거된다.
- [ ] 연결된 Rule의 Container 참조가 정리된다.
- [ ] Undo로 삭제를 취소할 수 있다.

### F-007 Container 이동

| 항목 | 내용 |
|------|------|
| **기능 ID** | F-007 |
| **기능명** | Container 이동 |
| **설명** | 사용자가 Container를 바탕화면의 다른 위치로 드래그하여 이동한다. |
| **목적** | 사용자가 바탕화면 레이아웃을 자유롭게 구성할 수 있게 한다. |
| **우선순위** | Must Have |
| **의존 기능** | F-004 (Container 생성) |

#### 상세 동작

1. 사용자가 Container의 제목 영역 또는 이동 핸들을 마우스로 클릭한다.
2. 드래그 시작 시: Container가 이동 모드로 전환된다 (시각적 피드백 표시).
3. 드래그 중: Container가 마우스 커서를 따라 실시간으로 이동한다.
   - Container 내부의 아이콘도 함께 이동한다.
   - 다른 Container와의 충돌(겹침) 여부를 실시간으로 표시한다.
4. 드래그 종료(마우스 버튼 해제) 시:
   - 현재 마우스 위치를 새로운 Container 위치로 확정한다.
   - 바탕화면 경계를 벗어나지 않도록 위치를 보정한다.
   - 새 위치를 설정 파일에 저장한다.
5. 이동 완료 후 Container 내 아이콘 위치를 새 Container 위치 기준으로 재계산한다.

#### 입력

| 항목 | 값 |
|------|-----|
| 드래그 시작 좌표 | 마우스 버튼 다운 시 좌표 |
| 드래그 종료 좌표 | 마우스 버튼 업 시 좌표 |
| 대상 Container ID | 이동할 Container |

#### 출력

| 항목 | 값 |
|------|-----|
| 새 Container 위치 | 확정된 (x, y) 좌표 |
| 저장 결과 | 성공 / 실패 |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| 바탕화면 경계 초과 | 경계 내 최대 허용 좌표로 위치 보정 |
| 드래그 중 화면 잠금 | 마지막 유효 위치로 Container 복귀 |
| 저장 실패 | 이동은 유지, 오류 로그 기록 |

#### 완료 기준 (Acceptance Criteria)

- [ ] Container를 드래그로 이동할 수 있다.
- [ ] 이동 중 Container 내 아이콘이 함께 이동한다.
- [ ] 바탕화면 경계를 벗어나지 않는다.
- [ ] 이동 완료 후 위치가 저장된다.
- [ ] 프로그램 재시작 후 이동된 위치가 유지된다.

- [ ] 변경된 크기가 저장된다.
- [ ] 프로그램 재시작 후 크기가 유지된다.

| 스타일 속성 | 옵션 |

| 배경색 | 색상 선택기 (RGB/HEX 입력 지원) |
| 배경 투명도 | 0~100% 슬라이더 |
| 테두리 색상 | 색상 선택기 |
| 테두리 두께 | 0~10px |
| 테두리 스타일 | 실선 / 점선 / 없음 |
| 제목 표시 | 표시 / 숨김 |
| 제목 글꼴 크기 | 소 / 중 / 대 |
| 제목 글꼴 색상 | 색상 선택기 |
| 모서리 둥글기 | 0~20px |

4. 스타일 변경 사항은 실시간으로 미리보기로 반영된다.
5. 확인 시 변경 사항이 Container에 적용되고 설정 파일에 저장된다.
6. 취소 시 변경 전 스타일로 복원된다.

#### 입력

| 항목 | 값 |
|------|-----|
| 대상 Container ID | 스타일 변경할 Container |
| 변경할 스타일 속성 및 값 | 사용자 입력 |

#### 출력

| 항목 | 값 |
|------|-----|
| 변경된 Container 스타일 | 적용된 스타일 속성 집합 |
| 시각적 갱신 | 바탕화면 내 Container 즉시 갱신 |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| 색상 값이 유효하지 않은 경우 | 입력 거부, 기존 값 유지 |
| 배경 투명도 100%로 설정 | 경고 메시지 ("Container가 보이지 않을 수 있습니다") |
| 저장 실패 | 스타일은 적용, 오류 로그 기록 |

#### 완료 기준 (Acceptance Criteria)

- [ ] 위 표의 모든 스타일 속성을 변경할 수 있다.
- [ ] 변경 사항이 실시간 미리보기로 표시된다.
- [ ] 취소 시 원래 스타일로 복원된다.
- [ ] 변경된 스타일이 저장되고 재시작 후 유지된다.

#### 비고

- 스타일 프리셋(테마) 기능은 향후 버전에서 고려한다.

### F-010 아이콘 자동 정렬

| 항목 | 내용 |
|------|------|
| **기능 ID** | F-010 |
| **기능명** | 아이콘 자동 정렬 |
| **설명** | Container 내 아이콘을 지정된 기준에 따라 자동으로 정렬하고 배치한다. |
| **목적** | Container 내 아이콘이 일관된 순서와 간격으로 배치되어 가독성을 높인다. |
| **우선순위** | Must Have |
| **의존 기능** | F-004 (Container 생성), F-001 (바탕화면 읽기) |

#### 상세 동작

1. 정렬 트리거 발생 시 (Container 생성, 아이콘 추가, 정렬 방식 변경, 크기 변경 등) 실행된다.
2. 지원하는 정렬 기준은 다음과 같다.

| 정렬 기준 | 설명 |
|-----------|------|
| 이름 오름차순 | 파일명 가나다/ABC 순 |
| 이름 내림차순 | 파일명 역순 |
| 확장자 | 확장자 기준 그룹화 후 이름 순 |
| 파일 유형 | 분류 유형 기준 (F-002 결과 활용) |
| 생성일 오름차순 | 오래된 파일 먼저 |
| 생성일 내림차순 | 최신 파일 먼저 |
| 수정일 오름차순 | 오래 전 수정된 파일 먼저 |
| 수정일 내림차순 | 최근 수정된 파일 먼저 |
| 수동 | 사용자가 직접 순서 지정 (F-011 연동) |

3. 정렬 방향은 좌→우, 위→아래(행 우선) 방식으로 배치한다.
4. 아이콘 간격은 설정에서 지정한 간격(픽셀)을 적용한다.
5. Container 너비를 초과하면 다음 행으로 줄바꿈한다.
6. 아이콘 위치 변경은 바탕화면 좌표 이동만 수행하며 실제 파일은 이동하지 않는다.

#### 입력

| 항목 | 값 |
|------|-----|
| Container ID | 정렬 대상 Container |
| 정렬 기준 | 선택된 정렬 방식 |
| Container 크기 | 현재 Container의 width, height |
| 아이콘 목록 | Container 내 아이콘 목록 |

#### 출력

| 항목 | 값 |
|------|-----|
| 재배치된 아이콘 좌표 | 각 아이콘의 새로운 (x, y) 좌표 |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| Container보다 아이콘이 많아 넘치는 경우 | 스크롤 제공 또는 Container 자동 확장 (설정에 따라) |
| 정렬 기준 속성이 없는 아이콘 존재 | 해당 속성 없는 아이콘은 목록 말미로 이동 |
| 바탕화면 좌표 이동 실패 | 오류 로그 기록, 기존 위치 유지 |

#### 완료 기준 (Acceptance Criteria)

- [ ] 위 9가지 정렬 기준이 모두 동작한다.
- [ ] 정렬 변경 시 즉시 재배치된다.
- [ ] 아이콘 수가 Container를 초과할 때 처리된다.
- [ ] 실제 파일은 이동되지 않는다.
- [ ] 정렬 후 아이콘 간격이 균일하게 유지된다.

### F-011 아이콘 순서 저장

| 항목 | 내용 |
|------|------|
| **기능 ID** | F-011 |
| **기능명** | 아이콘 순서 저장 |
| **설명** | 사용자가 수동으로 지정하거나 자동 정렬된 Container 내 아이콘의 순서를 저장한다. |
| **목적** | 프로그램 재시작 또는 바탕화면 갱신 이후에도 아이콘 순서가 유지되도록 한다. |
| **우선순위** | Must Have |
| **의존 기능** | F-010 (아이콘 자동 정렬), F-018 (설정 저장) |

#### 상세 동작

1. 저장 트리거: 아이콘 정렬 완료, 수동 순서 변경, 명시적 저장 요청.
2. 각 Container에 대해 현재 아이콘 순서(인덱스)를 기록한다.

| 저장 정보 | 설명 |
|-----------|------|
| Container ID | 소속 Container 식별자 |
| 아이콘 식별자 | 절대 경로 기반 |
| 순서 인덱스 | 0부터 시작하는 정수 |
| 저장 시각 | UTC 기준 타임스탬프 |

3. 저장 형식은 설정 파일 내 별도 섹션에 기록한다.
4. 프로그램 시작 시 저장된 순서를 불러와 Container 내 아이콘을 해당 순서로 배치한다.
5. 수동 정렬 모드에서 사용자가 아이콘을 드래그로 순서 변경 시 즉시 저장한다.

#### 입력

| 항목 | 값 |
|------|-----|
| Container ID | 순서를 저장할 Container |
| 아이콘 순서 목록 | 아이콘 식별자 + 인덱스 배열 |

#### 출력

| 항목 | 값 |
|------|-----|
| 저장 결과 | 성공 / 실패 |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| 저장된 아이콘이 실제로 존재하지 않는 경우 | 해당 항목 제거 후 나머지 순서 유지 |
| 저장 파일 쓰기 실패 | 오류 로그 기록, 메모리 내 순서는 유지 |
| 순서 정보 불일치(개수 다름) | 유효한 항목만 복원 후 나머지는 기본 순서 적용 |

#### 완료 기준 (Acceptance Criteria)

- [ ] Container 내 아이콘 순서가 저장된다.
- [ ] 프로그램 재시작 후 순서가 복원된다.
- [ ] 존재하지 않는 아이콘의 순서 정보는 자동 정리된다.
- [ ] 수동 순서 변경이 즉시 저장된다.

### F-012 Rule 생성

| 항목 | 내용 |
|------|------|
| **기능 ID** | F-012 |
| **기능명** | Rule 생성 |
| **설명** | 사용자가 새로운 Rule을 정의한다. Rule은 특정 조건을 만족하는 아이콘을 지정한 Container로 자동 배치하는 규칙이다. |
| **목적** | 반복적인 수동 정리 없이 새 파일을 자동으로 올바른 Container에 배치한다. |
| **우선순위** | Must Have |
| **의존 기능** | F-004 (Container 생성), F-018 (설정 저장) |

#### 상세 동작

1. 사용자가 Rule 생성 메뉴를 선택한다.
2. Rule 편집 다이얼로그가 열린다.
3. 사용자가 아래 항목을 설정한다.

**Rule 기본 정보**

| 항목 | 설명 |
|------|------|
| Rule ID | 자동 생성 (UUID) |
| Rule 이름 | 사용자 입력 (필수) |
| 활성화 여부 | 활성 / 비활성 (기본: 활성) |
| 우선순위 | 숫자 (낮을수록 높은 우선순위, F-015 연동) |

**조건(Condition) 설정 — 아래 중 하나 이상 선택**

| 조건 유형 | 상세 옵션 |
|-----------|-----------|
| 파일명 패턴 | 포함 / 시작 / 끝 / 정규식 |
| 확장자 | 특정 확장자 목록 (다중 선택) |
| 파일 유형 | F-002 분류 유형 선택 |
| 생성일 범위 | 날짜 범위 지정 |
| 수정일 범위 | 날짜 범위 지정 |

**조건 결합 방식**

| 옵션 | 설명 |
|------|------|
| AND | 모든 조건 만족 시 적용 |
| OR | 하나 이상 조건 만족 시 적용 |

**액션(Action) 설정**

| 항목 | 설명 |
|------|------|
| 대상 Container | 아이콘을 배치할 Container (필수, 기존 Container 선택) |

4. 저장 시 Rule 유효성 검사를 수행한다.
5. 유효한 경우 Rule을 저장하고 목록에 추가한다.
6. Rule 저장 즉시 기존 바탕화면 아이콘에 적용할지 사용자에게 묻는다.

#### 입력

| 항목 | 값 |
|------|-----|
| Rule 이름 | 사용자 입력 문자열 |
| 조건 목록 | 조건 유형 + 값 + 결합 방식 |
| 대상 Container ID | 액션 대상 Container |

#### 출력

| 항목 | 값 |
|------|-----|
| 새 Rule 객체 | 생성된 Rule의 모든 속성 |
| 저장 결과 | 성공 / 실패 |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| 이름이 빈 경우 | 저장 거부, 경고 메시지 |
| 조건이 하나도 없는 경우 | 저장 거부, 경고 메시지 |
| 대상 Container가 선택되지 않은 경우 | 저장 거부, 경고 메시지 |
| 대상 Container가 삭제된 경우 | Rule 비활성화 처리, 경고 로그 |
| 정규식이 유효하지 않은 경우 | 입력 거부, 오류 메시지 표시 |

#### 완료 기준 (Acceptance Criteria)

- [ ] Rule 이름, 조건, 대상 Container를 설정할 수 있다.
- [ ] 5가지 조건 유형을 모두 사용할 수 있다.
- [ ] AND/OR 조건 결합을 설정할 수 있다.
- [ ] 필수 항목 누락 시 저장이 거부된다.
- [ ] 생성된 Rule이 저장되고 목록에 표시된다.
- [ ] 저장 직후 기존 아이콘에 즉시 적용 여부를 선택할 수 있다.

### F-013 Rule 수정

| 항목 | 내용 |
|------|------|
| **기능 ID** | F-013 |
| **기능명** | Rule 수정 |
| **설명** | 사용자가 기존 Rule의 이름, 조건, 액션, 활성화 여부를 수정한다. |
| **목적** | 규칙이 변경될 때 새로 생성하지 않고 기존 Rule을 갱신할 수 있게 한다. |
| **우선순위** | Must Have |
| **의존 기능** | F-012 (Rule 생성), F-018 (설정 저장) |

#### 상세 동작

1. 사용자가 Rule 목록에서 수정할 Rule을 선택하고 수정 액션을 실행한다.
2. F-012와 동일한 Rule 편집 다이얼로그가 열리며 기존 값이 채워진 상태로 표시된다.
3. 사용자가 원하는 항목을 변경한다.
4. 저장 시 동일한 유효성 검사를 수행한다.
5. 저장 완료 후 수정된 Rule을 기존 아이콘에 재적용할지 사용자에게 묻는다.
6. 재적용 선택 시: 현재 바탕화면의 모든 비제외 아이콘에 수정된 Rule을 적용한다.
7. 수정 전 상태를 Undo 스택에 저장한다.

#### 입력

| 항목 | 값 |
|------|-----|
| 대상 Rule ID | 수정할 Rule 식별자 |
| 변경된 속성 및 값 | 사용자 입력 |

#### 출력

| 항목 | 값 |
|------|-----|
| 수정된 Rule 객체 | 갱신된 Rule |
| 저장 결과 | 성공 / 실패 |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| 유효성 검사 실패 | F-012 예외 처리와 동일 |
| 저장 실패 | 수정 사항 롤백, 오류 로그 기록 |

#### 완료 기준 (Acceptance Criteria)

- [ ] 기존 Rule의 모든 속성을 수정할 수 있다.
- [ ] 수정 다이얼로그에 기존 값이 사전 입력된 상태로 표시된다.
- [ ] 수정 후 기존 아이콘 재적용 여부를 선택할 수 있다.
- [ ] 수정 사항이 저장된다.
- [ ] Undo로 수정 전 상태로 되돌릴 수 있다.

### F-014 Rule 삭제

| 항목 | 내용 |
|------|------|
| **기능 ID** | F-014 |
| **기능명** | Rule 삭제 |
| **설명** | 사용자가 더 이상 필요하지 않은 Rule을 삭제한다. |
| **목적** | 불필요한 Rule을 제거하여 Rule 목록을 간결하게 유지한다. |
| **우선순위** | Must Have |
| **의존 기능** | F-012 (Rule 생성), F-018 (설정 저장) |

#### 상세 동작

1. 사용자가 Rule 목록에서 삭제할 Rule을 선택하고 삭제 액션을 실행한다.
2. 삭제 확인 다이얼로그를 표시한다.
   - 안내 문구: "이 Rule을 삭제하면 더 이상 자동으로 적용되지 않습니다. 기존 배치된 아이콘 위치는 변경되지 않습니다."
3. 사용자 확인 시:
   1. Rule 객체를 목록에서 제거한다.
   2. 설정 파일에서 해당 Rule 정보를 삭제한다.
   3. 해당 Rule과 연결된 Container는 영향을 받지 않는다.
4. 사용자 취소 시: 아무 동작도 하지 않는다.

#### 입력

| 항목 | 값 |
|------|-----|
| 대상 Rule ID | 삭제할 Rule 식별자 |
| 사용자 확인 | 확인 또는 취소 |

#### 출력

| 항목 | 값 |
|------|-----|
| 삭제 결과 | 성공 / 취소 / 실패 |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| 설정 파일 쓰기 실패 | 오류 로그 기록, Rule은 목록에서 제거하되 재시작 시 복원될 수 있음을 알림 |

#### 완료 기준 (Acceptance Criteria)

- [ ] Rule 삭제 시 확인 다이얼로그가 표시된다.
- [ ] 삭제 후 Rule이 목록에서 제거된다.
- [ ] 삭제된 Rule은 이후 새 파일에 적용되지 않는다.
- [ ] 기존 배치된 아이콘 위치는 변경되지 않는다.
- [ ] 삭제 정보가 설정 파일에 반영된다.

### F-015 Rule 우선순위 관리

| 항목 | 내용 |
|------|------|
| **기능 ID** | F-015 |
| **기능명** | Rule 우선순위 관리 |
| **설명** | 여러 Rule이 동일한 아이콘에 매칭될 때 어떤 Rule을 우선 적용할지 순서를 관리한다. |
| **목적** | Rule 충돌 시 예측 가능하고 사용자가 제어할 수 있는 방식으로 처리한다. |
| **우선순위** | Must Have |
| **의존 기능** | F-012 (Rule 생성) |

#### 상세 동작

1. Rule 목록 화면에서 우선순위 순서대로 Rule이 나열된다.
2. 사용자가 Rule을 드래그하여 우선순위 순서를 변경할 수 있다.
3. 우선순위 숫자가 낮을수록(1이 최고 우선순위) 먼저 적용된다.
4. 아이콘에 Rule을 적용하는 순서:
   1. 활성화된 Rule을 우선순위 순으로 정렬한다.
   2. 우선순위 1번 Rule부터 순서대로 아이콘에 조건 매칭을 시도한다.
   3. 첫 번째 매칭되는 Rule의 액션을 실행한다.
   4. 이후 Rule은 평가하지 않는다 (First Match 방식).
5. 비활성화된 Rule은 순서와 무관하게 매칭 대상에서 제외된다.
6. 우선순위 변경 즉시 설정 파일에 저장된다.

#### 입력

| 항목 | 값 |
|------|-----|
| Rule 목록 | 현재 등록된 모든 Rule |
| 새 순서 | 사용자가 드래그로 지정한 순서 |

#### 출력

| 항목 | 값 |
|------|-----|
| 재정렬된 Rule 목록 | 새 우선순위 순서로 정렬된 Rule 목록 |
| 저장 결과 | 성공 / 실패 |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| 어떤 Rule도 매칭되지 않는 아이콘 | 기존 위치 유지, 정렬 대상에서 제외 |
| 저장 실패 | 오류 로그 기록, 현재 세션에는 변경된 순서 유지 |

#### 완료 기준 (Acceptance Criteria)

- [ ] Rule 목록에서 드래그로 순서를 변경할 수 있다.
- [ ] 동일 아이콘에 복수 Rule 매칭 시 First Match 방식으로 처리된다.
- [ ] 비활성 Rule은 매칭에서 제외된다.
- [ ] 변경된 우선순위가 저장된다.

### F-016 실시간 바탕화면 감시

| 항목 | 내용 |
|------|------|
| **기능 ID** | F-016 |
| **기능명** | 실시간 바탕화면 감시 |
| **설명** | 바탕화면 폴더의 파일 시스템 변경 이벤트를 실시간으로 감지한다. |
| **목적** | 사용자 개입 없이 새 파일 추가, 파일 삭제, 파일 이름 변경을 즉시 감지하여 자동화 처리를 가능하게 한다. |
| **우선순위** | Must Have |
| **의존 기능** | F-001 (바탕화면 읽기), F-017 (새 파일 자동 정리) |

#### 상세 동작

1. 프로그램 시작 시 Windows FileSystemWatcher (또는 동등한 API)를 사용하여 바탕화면 경로를 감시 등록한다.
2. 감시 대상 이벤트:

| 이벤트 유형 | 설명 |
|-------------|------|
| 파일 생성 (Created) | 새 파일 또는 폴더가 바탕화면에 생성됨 |
| 파일 삭제 (Deleted) | 파일 또는 폴더가 바탕화면에서 제거됨 |
| 파일 이름 변경 (Renamed) | 파일 또는 폴더 이름이 변경됨 |
| 파일 이동 (Moved In) | 다른 위치에서 바탕화면으로 파일이 이동됨 |
| 파일 이동 (Moved Out) | 바탕화면에서 다른 위치로 파일이 이동됨 |

3. 이벤트 발생 시 해당 이벤트를 내부 이벤트 큐에 추가한다.
4. 이벤트 큐는 500ms 간격으로 처리하여 연속 이벤트를 배치 처리한다 (디바운싱).
5. 처리된 이벤트는 F-017 (새 파일 자동 정리) 로 전달된다.
6. 감시는 프로그램이 실행 중인 동안 지속적으로 동작한다.
7. 사용자 설정에서 감시 기능을 일시 중지/재개할 수 있다.

#### 입력

| 항목 | 값 |
|------|-----|
| 감시 경로 | 사용자 바탕화면 + 공용 바탕화면 |
| 파일 시스템 이벤트 | OS로부터 전달되는 이벤트 |

#### 출력

| 항목 | 값 |
|------|-----|
| 처리된 이벤트 목록 | 유형 + 대상 파일 정보 |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| 감시 등록 실패 | 오류 로그 기록, 주기적 폴링(5초)으로 대체 |
| 이벤트 큐 과부하 | 오래된 이벤트부터 처리, 큐 최대 크기 제한 |
| 감시 중 경로 접근 불가 | 감시 재등록 시도, 실패 시 사용자 알림 |

#### 완료 기준 (Acceptance Criteria)

- [ ] 바탕화면에 파일이 추가되면 즉시 (500ms 이내) 감지된다.
- [ ] 파일 삭제, 이름 변경이 감지된다.
- [ ] 연속 이벤트가 디바운싱으로 묶여 처리된다.
- [ ] 감시 기능을 설정에서 일시 중지/재개할 수 있다.
- [ ] 감시 실패 시 폴링으로 자동 대체된다.

### F-017 새 파일 자동 정리

| 항목 | 내용 |
|------|------|
| **기능 ID** | F-017 |
| **기능명** | 새 파일 자동 정리 |
| **설명** | F-016에서 감지된 새 파일을 등록된 Rule에 따라 자동으로 적절한 Container로 배치한다. |
| **목적** | 사용자가 수동으로 아이콘을 정리하지 않아도 새 파일이 자동으로 올바른 Container에 배치되게 한다. |
| **우선순위** | Must Have |
| **의존 기능** | F-016 (실시간 바탕화면 감시), F-015 (Rule 우선순위 관리), F-010 (아이콘 자동 정렬) |

#### 상세 동작

1. F-016에서 "파일 생성" 또는 "파일 이동(Moved In)" 이벤트를 수신한다.
2. 새 파일 정보를 수집한다 (F-001 방식 동일).
3. F-003 기준으로 기본 제외 아이콘인지 확인한다. 해당할 경우 처리를 중단한다.
4. F-002를 통해 파일 유형을 분류한다.
5. F-015에 정의된 우선순위 순으로 활성 Rule을 순회하며 조건 매칭을 수행한다.
6. 첫 번째 매칭 Rule이 발견되면:
   1. 해당 Rule의 대상 Container를 확인한다.
   2. 아이콘을 Container 내 빈 자리에 배치한다.
   3. Container 내 정렬 규칙에 따라 재정렬한다 (F-010).
   4. 배치 결과를 로그에 기록한다 (F-022).
7. 매칭되는 Rule이 없으면 아이콘을 현재 위치에 그대로 유지한다.
8. 파일 삭제 이벤트 수신 시: 해당 아이콘을 내부 목록에서 제거하고 Container에서 해제한다.
9. 파일 이름 변경 이벤트 수신 시: 아이콘 정보를 갱신하고 Rule을 재평가한다.

#### 입력

| 항목 | 값 |
|------|-----|
| 파일 시스템 이벤트 | F-016에서 전달된 이벤트 |
| 활성 Rule 목록 | F-015 기준 우선순위 정렬된 목록 |

#### 출력

| 항목 | 값 |
|------|-----|
| 배치 결과 | 아이콘이 배치된 Container ID + 위치 |
| 처리 로그 | F-022 연동 |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| 대상 Container가 존재하지 않는 경우 | Rule 비활성화, 오류 로그 기록, 아이콘 현재 위치 유지 |
| Container 내 공간 부족 | Container 자동 확장 또는 스크롤 처리 (설정 기반) |
| 아이콘 좌표 이동 실패 | 오류 로그 기록, 수동 처리 안내 |
| 파일 생성 직후 접근 불가 (잠금) | 200ms 후 재시도, 최대 3회 |

#### 완료 기준 (Acceptance Criteria)

- [ ] 새 파일이 생성되면 자동으로 Rule이 평가된다.
- [ ] 매칭된 Rule의 Container로 아이콘이 이동된다.
- [ ] 매칭 Rule이 없으면 아이콘 위치가 그대로 유지된다.
- [ ] 기본 제외 아이콘은 자동 정리되지 않는다.
- [ ] 파일 삭제 시 내부 목록이 갱신된다.
- [ ] 파일 이름 변경 시 Rule이 재평가된다.
- [ ] 모든 처리 결과가 로그에 기록된다.

### F-019 설정 불러오기

- [ ] 프로그램 시작 시 설정이 자동으로 불러와진다.

- [ ] 파일 없을 시 기본값으로 정상 동작한다.
- [ ] 버전 마이그레이션이 자동으로 수행된다.

| **의존 기능** | F-004~F-011 (Container 관련 기능), F-018 (설정 저장) |

#### 상세 동작

1. 사용자가 Layout 저장 메뉴를 선택한다.
2. Layout 이름 입력 다이얼로그를 표시한다.
3. 사용자가 이름을 입력하고 확인한다.
4. 현재 상태를 스냅샷으로 캡처한다:

| 저장 항목 | 설명 |
|-----------|------|
| Layout ID | UUID |
| Layout 이름 | 사용자 입력 |
| 저장 시각 | UTC 타임스탬프 |
| Container 목록 | 각 Container의 ID, 이름, 위치, 크기, 스타일 |
| 아이콘 배치 | Container별 아이콘 순서 및 위치 |
| 바탕화면 해상도 | 저장 시점의 화면 해상도 |

5. Layout 데이터를 별도 파일로 저장한다: `%APPDATA%\DesktopOrganizer\layouts\{layout_id}.json`
6. Layout 목록 인덱스 파일에 새 Layout 정보를 추가한다.

#### 입력

| 항목 | 값 |
|------|-----|
| Layout 이름 | 사용자 입력 |
| 현재 상태 | 모든 Container + 아이콘 배치 정보 |

#### 출력

| 항목 | 값 |
|------|-----|
| Layout 파일 | `%APPDATA%\DesktopOrganizer\layouts\{id}.json` |
| 저장 결과 | 성공 / 실패 |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| 이름 중복 | 경고 후 덮어쓰기 또는 다른 이름 입력 선택 |
| 이름 빈 문자열 | 저장 거부 |
| 저장 실패 | 오류 로그, 사용자 알림 |
| 최대 Layout 수 초과 | 경고 메시지, 기존 Layout 삭제 후 저장 안내 |

#### 완료 기준 (Acceptance Criteria)

- [ ] Layout 이름을 지정하여 저장할 수 있다.
- [ ] 저장 시점의 모든 Container와 아이콘 배치가 캡처된다.
- [ ] 저장된 Layout이 목록에 표시된다.
- [ ] 동일 이름 덮어쓰기 시 확인 절차가 있다.

#### 비고

- 최대 저장 Layout 수는 초기 버전 기준 설정 필요 (예: 10개).

### F-021 Layout 복원

| 항목 | 내용 |
|------|------|
| **기능 ID** | F-021 |
| **기능명** | Layout 복원 |
| **설명** | 저장된 Layout을 선택하여 바탕화면을 해당 Layout의 상태로 복원한다. |
| **목적** | 저장된 바탕화면 구성으로 즉시 전환하여 작업 효율을 높인다. |
| **우선순위** | Should Have |
| **의존 기능** | F-020 (Layout 저장), F-004~F-011 (Container 관련 기능) |

#### 상세 동작

1. 사용자가 Layout 목록에서 복원할 Layout을 선택한다.
2. 복원 확인 다이얼로그를 표시한다.
   - 안내 문구: "현재 Layout이 선택한 Layout으로 교체됩니다. 저장되지 않은 변경 사항이 있으면 먼저 저장하세요."
3. 사용자 확인 시:
   1. 현재 바탕화면 상태를 자동으로 임시 저장한다.
   2. Layout 파일을 읽는다.
   3. 해상도 불일치 여부를 확인한다.
   4. Container 목록을 복원한다 (기존 Container 교체).
   5. 아이콘 배치를 복원한다.
   6. 저장된 아이콘이 현재 바탕화면에 없는 경우 해당 항목을 건너뛴다.
4. 사용자 취소 시: 아무 동작도 하지 않는다.

#### 입력

| 항목 | 값 |
|------|-----|
| 대상 Layout ID | 복원할 Layout 식별자 |
| 사용자 확인 | 확인 또는 취소 |

#### 출력

| 항목 | 값 |
|------|-----|
| 복원 결과 | 성공 / 부분 성공 / 실패 |
| 누락된 아이콘 목록 | 복원 시 현재 바탕화면에 없는 아이콘 목록 |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| Layout 파일 손상 | 오류 로그, 복원 중단, 사용자 알림 |
| 해상도 불일치 | 경고 메시지 후 비율 기반으로 Container 위치/크기 조정 |
| 저장된 아이콘이 현재 없는 경우 | 해당 아이콘 건너뛰고 나머지 복원, 누락 목록 표시 |
| Container 복원 실패 | 오류 로그, 해당 Container만 건너뜀 |

#### 완료 기준 (Acceptance Criteria)

- [ ] Layout 목록에서 복원할 Layout을 선택할 수 있다.
- [ ] 복원 전 확인 다이얼로그가 표시된다.
- [ ] 복원 전 현재 상태가 임시 저장된다.
- [ ] Container와 아이콘 배치가 복원된다.
- [ ] 존재하지 않는 아이콘은 건너뛰고 나머지는 정상 복원된다.
- [ ] 해상도 불일치 시 위치가 비율 기반으로 조정된다.

### F-022 로그 기능

| 항목 | 내용 |
|------|------|
| **기능 ID** | F-022 |
| **기능명** | 로그 기능 |
| **설명** | 프로그램의 주요 동작, 오류, 경고를 로컬 로그 파일에 기록한다. |
| **목적** | 문제 발생 시 원인을 추적하고, 자동화 동작 이력을 사용자가 확인할 수 있게 한다. |
| **우선순위** | Should Have |
| **의존 기능** | 없음 (기반 기능, 모든 기능에서 활용) |

#### 상세 동작

1. 프로그램 시작 시 로그 시스템을 초기화한다.
2. 로그 파일 경로: `%APPDATA%\DesktopOrganizer\logs\desktop_organizer_YYYYMMDD.log`
3. 날짜별로 로그 파일을 분리한다.
4. 로그 레벨:

| 레벨 | 용도 |
|------|------|
| INFO | 정상 동작 이력 (아이콘 배치 완료, 설정 저장 완료 등) |
| WARN | 비정상이지만 계속 동작 가능한 상황 |
| ERROR | 기능 실패 또는 예외 상황 |
| DEBUG | 개발/진단 목적 상세 정보 (기본 비활성화) |

5. 각 로그 항목에 포함할 정보:

| 항목 | 예시 |
|------|------|
| 타임스탬프 | 2026-06-18 14:23:01.123 UTC |
| 레벨 | INFO |
| 기능 ID | F-017 |
| 메시지 | "아이콘 'report.pdf'를 Container 'Documents'에 배치 완료" |

6. 로그 파일 보존 기간: 최대 30일 (오래된 파일 자동 삭제).
7. 로그 파일 최대 크기: 10MB (초과 시 롤링).
8. 사용자는 설정에서 로그 레벨을 변경할 수 있다 (INFO / WARN+ERROR만 / 비활성화).
9. 사용자가 로그 파일을 열어볼 수 있는 메뉴를 제공한다.

#### 입력

| 항목 | 값 |
|------|-----|
| 로그 레벨 | INFO / WARN / ERROR / DEBUG |
| 기능 ID | 호출 기능 식별자 |
| 메시지 | 로그 내용 문자열 |

#### 출력

| 항목 | 값 |
|------|-----|
| 로그 파일 | 날짜별 텍스트 파일 |

#### 예외 처리

| 예외 상황 | 처리 방법 |
|-----------|-----------|
| 로그 파일 쓰기 실패 | 메모리 내 버퍼에 임시 보관, 재시도 후 계속 실패 시 로깅 중단 (프로그램은 계속 동작) |
| 디스크 공간 부족 | 가장 오래된 로그 파일부터 삭제 후 재시도 |
| 로그 디렉터리 생성 실패 | 오류를 출력으로 표시하고 로깅 비활성화 |

#### 완료 기준 (Acceptance Criteria)

- [ ] 날짜별로 로그 파일이 생성된다.
- [ ] INFO, WARN, ERROR, DEBUG 4가지 레벨이 지원된다.
- [ ] 각 로그 항목에 타임스탬프, 레벨, 기능 ID, 메시지가 포함된다.
- [ ] 30일 이후 로그 파일이 자동 삭제된다.
- [ ] 10MB 초과 시 롤링된다.
- [ ] 사용자가 설정에서 로그 레벨을 변경할 수 있다.
- [ ] 사용자가 메뉴에서 로그 파일을 열 수 있다.
- [ ] 로그 실패 시 프로그램이 계속 동작한다.
